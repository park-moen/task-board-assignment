# AI 도구 활용 (AI_USAGE)

> AI 사용은 **권장**됩니다. 이 문서는 "얼마나 썼는지"가 아니라
> **AI가 준 답을 어떻게 검증·판단했는지**를 봅니다. 솔직하게 적어 주세요.

## 1. 사용한 도구
- Claude Code

## 2. 주로 어디에 썼나
| 작업 | 사용한 도구 | 활용 방식 |
|------|-------------|-----------|
| 테스트 코드 작성 | Claude Code | 로컬에서 발견한 버그(타이밍 의존적 경쟁 상태 등)를 설명하고, 결정적으로 재현 가능한 테스트 코드 생성 요청 |

## 3. AI 답을 그대로 쓰지 않고 검증·거부·수정한 사례 (중요)

**사례 1 — 로드 상태의 `retry: false` 근거 재검토**
- 로드 상태 처리([Issue #5](https://github.com/park-moen/task-board-assignment/issues/5))에서 에러 상태를 어떻게 노출할지 설계하던 중, Claude가 `retry: false`를 제안하며 두 가지 근거를 함께 제시했습니다: ① 에러 상태가 요구사항이자 검증 대상이므로 결정적으로 노출되어야 한다, ② GET 요청은 실패율이 낮아 자동 재시도의 실익이 적다. ②에 대해 "대부분 일시적 오류라 성공 확률이 높다"는 초기 반박은 실패의 성질을 확인 없이 일반화한 것이라 보고, 대신 `src/mocks/handlers.ts`의 실패 조건(`Math.random() < READ_FAILURE_RATE`)이 매 요청 독립적으로 실행됨을 코드로 확인해 "실패율이 낮으면 재시도가 (1 - READ_FAILURE_RATE)의 확률로 성공하므로 오히려 유지하는 편이 유리하다"는 근거로 인과관계가 반대임을 지적했습니다. 이 지적을 반영해 ②는 근거에서 제외하고 ①만을 근거로 선택했습니다.
- 이후 [TanStack Query 공식 문서(Important Defaults)](https://tanstack.com/query/latest/docs/framework/react/guides/important-defaults)에서 "쿼리가 실패하면 오류를 UI에 표시하기 전에 재시도가 3회 실행된다"는 문장을 확인하고, `READ_FAILURE_RATE`를 최대로 올려 재시도가 모두 실행된 뒤 실제로 에러 상태가 뜨는지 직접 실행해 검증했습니다. 그 결과 ①의 전제("자동 재시도가 에러 상태를 가린다")도 성공한 재시도에만 해당하고, 실패가 계속되면 재시도 실행 후 결국 에러 상태가 뜬다는(단지 몇 초 지연될 뿐) 사실을 확인했습니다. 이를 반영해 `retry: false` 결정 자체를 번복하고, 기본 재시도(3회)를 유지하는 쪽으로 최종 선택했습니다.

**사례 2 — 낙관적 업데이트 롤백 범위 버그**
- 낙관적 업데이트를 구현할 때, 초기에는 Claude가 제안한 대로 `onMutate`에서 캐시 전체(`previousTasks`)를 스냅샷으로 저장했다가 실패 시 전체를 복원하는 방식으로 작성했습니다. 타입체크와 기존 유닛 테스트는 모두 통과했습니다.
- 하지만 `WRITE_FAILURE_RATE`를 0.5로 올린 뒤, 로컬에서 카드를 여러 개 빠르게 연속으로 이동시키며 직접 테스트하던 중, 카드 A의 이동이 실패해 롤백되면서 그 사이 이미 성공한 카드 B의 변경까지 원래대로 되돌아가는 것을 발견했습니다. 원인은 A의 `onMutate` 시점에 찍은 스냅샷에 B의 이전 상태까지 포함되어 있어, A가 실패했을 때 그 스냅샷 전체로 복원하면 B의 최신 성공 결과까지 되돌려버리기 때문이었습니다.
- 이 버그는 타입체크나 기존 유닛 테스트로는 잡히지 않고, 여러 카드를 동시에 조작하는 실제 시나리오를 직접 실행해봐야만 드러나는 문제였습니다. 다만 실제 네트워크 지연에 의존하는 로컬 재현은 타이밍이 매번 달라 회귀 테스트로 남기기 어렵다고 판단해, Claude에게 문제 상황을 설명하고 `deferred` 프로미스로 응답 순서를 결정적으로 제어하는 테스트 코드(`mutations.test.tsx`)를 생성받아 실패를 먼저 재현했습니다. 이후 롤백 스냅샷 범위를 배열 전체(`previousTasks`)에서 이동시킨 카드 하나(`previousTask`)로 좁혀 직접 수정했고, 테스트 통과로 재검증했습니다.

**사례 3 — `scope` 해결책의 숨은 결함들**
- Issue #8(경쟁 상태) 해결책으로 Claude가 TanStack Query의 `scope` 옵션(같은 리소스에 대한 뮤테이션을 직렬 실행)을 제안하며, 이것으로 README의 요구사항("늦게 도착한 오래된 응답이 최신 상태를 덮어쓰지 않아야 함")이 해결된다고 답했습니다.
- 이 답을 그대로 받아들이지 않고 "정확히 해결하는지" 재확인을 요청했고, 함께 재현 테스트를 짜본 결과 두 가지 숨은 결함을 발견했습니다: ① 직렬화된 두 번째 요청이 dispatch 시점에 캡처한 낡은 `version`을 그대로 사용해 409 충돌이 발생하는 문제, ② `onSuccess`가 태스크 전체를 교체해서, 먼저 보낸 요청의 응답이 뒤늦게 도착했을 때 이미 반영된 더 최신 낙관적 상태를 도로 덮어쓰는(깜빡이는) 문제.
- 두 문제 모두 재현 테스트로 먼저 확인한 뒤 수정(`mutationFn`에서 `version` 재조회, `onSuccess`에서 `version`/`updatedAt`만 병합)했습니다.
- 구현 단계에서는 Claude가 처음 제안한 `scope: { id: 'move-task' }`(고정 문자열)도 검증 없이 그대로 썼다면, 카드가 달라도 전부 같은 scope로 묶여 서로 다른 카드끼리도 불필요하게 직렬화되는 문제가 있었습니다. TanStack Query 소스코드(`mutationObserver.ts`)를 직접 확인해 `scope`가 훅 생성 시점에 고정되고 호출마다 바뀔 수 없다는 걸 확인한 뒤, `queryClient.getMutationCache().build()`로 호출마다 카드 id를 scope로 부여하도록 직접 수정했습니다.
- 이 두 가지를 고친 뒤에도, `WRITE_FAILURE_RATE`를 0.5로 올리고 브라우저에서 카드를 여러 번 빠르게 연속 이동시키는 수동 테스트를 진행하던 중 간헐적으로 409 충돌이 발생하는 걸 발견했습니다(Network 탭에서 같은 카드에 대해 서로 다른 `version`의 실패한 PATCH 요청을 확인). 재현 테스트로 원인을 좁혀보니, `onSuccess`는 부분 병합으로 고쳤지만 `onError`는 여전히 스냅샷 전체(`previousTask`, `version` 포함)로 복원하고 있었습니다. 그래서 한 요청이 실패하면, 그 사이 다른 요청이 이미 올려둔 `version`까지 되돌아가 다음 요청이 낡은 `version`을 보내는 문제였습니다. `onSuccess`와 같은 원칙으로 `onError`도 `status`만 되돌리고 `version`은 건드리지 않도록 수정했습니다.


## 4. 스스로 판단해서 결정한 부분
- AI 제안과 다르게 결정한 지점과 그 이유.
- 프로젝트 폴더 구조를 FSD/DDD처럼 기능(도메인) 단위로 나눌지, 아니면 기술적 역할 단위로 나눌지를 스스로 판단해서 결정했습니다. 이 프로젝트는 도메인이 "태스크" 하나뿐이라 기능별로 슬라이스를 나눠도 얻을 실익이 없다고 보고, `api/`(서버 통신)·`lib/`(순수 로직)·`components/`(UI)처럼 역할 단위 구조를 택했습니다(`CLAUDE.md`에 근거 기록). Toast 기능을 추가할 때 이 원칙을 다시 검증할 기회가 있었는데, Toast 관련 코드를 기능 단위로 `components/Toast/`에 묶는 대신 새로운 역할 계층(`contexts/`)을 신설해 기존 구조와의 일관성을 유지하는 쪽을 택했습니다. 이 과정에서 "계층별"이라는 표현이 FSD의 layer+slice 이중 축과 혼동될 수 있다는 것도 스스로 짚어내, "역할별"이라는 더 정확한 용어로 재정리했습니다.
- Toast 컴포넌트(`src/contexts/ToastContext.tsx`) 구현 중 `eslint-plugin-react-refresh`의 `only-export-components` 경고가 발생했습니다. Claude는 Context/Provider와 `useToast` 훅을 별도 파일로 분리해 Vite Fast Refresh(HMR)를 유지하는 방안을 제안했지만, 두 가지 근거로 제안과 다르게 한 파일에 유지하기로 했습니다. ① 이 파일은 76줄이고 export는 3개(`ToastProvider`, `useToast`, `ToastContainer`)라, 3개 파일로 분리하면 코드량에 비해 파일을 오가며 참조하는 탐색 비용이 불균형하게 늘어납니다. ② Fast Refresh가 깨져 전체 새로고침이 발생해도 잃는 상태는 화면에 렌더링된 Toast 배열(휘발성 UI 상태)뿐이라, 손실 비용이 낮습니다. 이 co-location 방식은 앞으로 비슷한 상황(관련 상태·훅·컴포넌트를 한 파일에 모으는 경우)에서 우선적으로 채택할 방침이라, 그때마다 `eslint-disable-next-line` 주석을 반복해서 다는 대신 `eslint.config.js`에서 규칙 자체를 프로젝트 전체적으로 껐습니다(파일별 예외 처리가 아니라, 반복될 패턴에 대한 의도적인 전역 선택). 다만 이 판단은 현재까지의 경험을 기록한 것으로, 이후 다른 상황을 겪으며 방침이 바뀔 수 있습니다.
